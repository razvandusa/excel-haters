WORKSPACE FILE EXPLANATION

This document explains what each file in this workspace does, how the files relate
to each other, and what role they play in the current app.

The workspace has two layers:

1. Workspace root files
   These are helper files for the repository itself.

2. The actual frontend app inside `excel-haters/`
   This is the Vite + React application that runs in the browser.


========================================================================
1. ROOT-LEVEL WORKSPACE FILES
========================================================================

File: `AGENTS.md`

Purpose:
- This is the local instruction file for coding agents working in this repo.
- It describes the app structure, conventions, and the current state of the app.

What it currently tells an agent:
- The app is a Vite + React SPA.
- Styling is split between JSX structure and shared classes in `src/index.css`.
- The configurator feature is the most developed part of the application.
- Terminals come from the backend API.
- Components on the main configurator page come from local JSON.
- The terminal detail page shows component columns grouped by type.

Why it matters:
- It is the fastest place to understand project expectations before editing code.
- It reduces guesswork when adding new files or features.


File: `README.md`

Purpose:
- This is the root readme for the workspace.

Current state:
- It is empty.

Why it matters:
- Right now it does not help the project.
- It could later be used for top-level setup instructions if this workspace starts
  containing more than one app or service.


File: `notes.txt`

Purpose:
- This is a scratchpad of backend endpoints and feature ideas.
- It acts like informal product and API notes rather than production code.

What it contains:
- Sample `curl` commands for terminals and terminal components.
- Notes for the Configurator tab API.
- Notes for future Timetables, Flights, and Recommendation tabs.

Why it matters:
- It documents the backend contract the frontend is expected to use.
- It explains future feature direction even though many of those screens are still
  placeholders.


File: `package-lock.json`

Purpose:
- This is a lockfile at the workspace root, not inside the Vite app.

Current state:
- It is almost empty and only identifies the root package as `client-admin`.
- It does not currently drive the React app inside `excel-haters/`.

Why it matters:
- It suggests someone ran npm in the root at some point.
- The actual frontend dependency lock that matters for the app is
  `excel-haters/package-lock.json`.


========================================================================
2. APP ROOT: `excel-haters/`
========================================================================

This folder contains the real frontend application.


File: `excel-haters/package.json`

Purpose:
- This is the main npm manifest for the frontend app.

What it defines:
- Project name: `excel-haters`
- Scripts:
  - `npm run dev` starts the Vite dev server.
  - `npm run build` creates a production bundle.
  - `npm run lint` runs ESLint.
  - `npm run preview` serves the built app locally.
- Runtime dependencies:
  - `react`
  - `react-dom`
  - `react-router-dom`
- Dev dependencies:
  - Vite
  - Tailwind integration for Vite
  - ESLint and React-related lint plugins

Why it matters:
- This file defines how the app is started, built, and checked.


File: `excel-haters/package-lock.json`

Purpose:
- This is the real dependency lockfile for the frontend app.

What it does:
- Pins exact package versions resolved by npm.
- Makes installs reproducible across machines.

Why it matters:
- It prevents "works on my machine" dependency drift.
- It is generated data, not hand-authored application logic.


File: `excel-haters/vite.config.js`

Purpose:
- This configures Vite for local development and production builds.

What it currently does:
- Enables the React plugin.
- Enables the Tailwind Vite plugin.
- Defines a dev server proxy:
  - Requests beginning with `/api`
  - Are proxied to `http://10.1.0.135:8080`

Why the proxy matters:
- The browser calls `/api/terminals` instead of calling the backend host directly.
- This avoids common CORS problems in local development.

Important limitation:
- This proxy exists in the Vite dev server only.
- It does not exist automatically in a production deployment.


File: `excel-haters/eslint.config.js`

Purpose:
- This defines lint rules for the frontend source code.

What it does:
- Starts from ESLint's recommended JS rules.
- Adds React Hooks rules.
- Adds the React Refresh/Vite rule set.
- Applies to `*.js` and `*.jsx` files.
- Ignores the `dist` directory.
- Allows unused variables only when they follow the configured ignore rule.

Why it matters:
- It catches mistakes early.
- It helps keep the codebase consistent.


File: `excel-haters/index.html`

Purpose:
- This is the HTML shell loaded by the browser.

What it does:
- Declares the document structure.
- Creates the `<div id="root"></div>` mount point.
- Loads the React app through `/src/main.jsx`.

Why it matters:
- React does not render directly to the whole HTML document.
- It mounts inside this root container.


File: `excel-haters/README.md`

Purpose:
- This is the Vite app's readme.

Current state:
- It is still the default React + Vite template readme.

What it describes:
- Basic React + Vite setup.
- The plugin choices.
- Notes about React Compiler and ESLint expansion.

Why it matters:
- It is mostly template material right now.
- It does not describe the airport admin app specifically.


File: `excel-haters/public/vite.svg`

Purpose:
- Static asset from the Vite template.

What it does:
- Can be served directly from the public folder.

Current relevance:
- It is used as the site favicon via `index.html`.


File: `excel-haters/src/assets/react.svg`

========================================================================
3. APPLICATION ENTRY AND ROUTING
========================================================================

File: `excel-haters/src/main.jsx`

Purpose:
- This is the JavaScript entry point for the entire React app.

What it does:
- Imports React strict mode.
- Imports ReactDOM's `createRoot`.
- Imports `BrowserRouter` from `react-router-dom`.
- Imports global styles from `index.css`.
- Imports the root `App` component.
- Renders the whole application into `#root`.

Why it matters:
- Without this file, the browser would load HTML but no React UI.
- It is where routing support is enabled for the entire app.


File: `excel-haters/src/App.jsx`

Purpose:
- This is the root app shell and route switcher.

What it defines:
- The top navigation tab list.
- The route table for every top-level screen.

Routes currently defined:
- `/` redirects to `/configurator`
- `/configurator` shows the main configurator dashboard
- `/configurator/:terminalName` shows the terminal detail page
- `/timetables` shows the placeholder timetables page
- `/flights` shows the placeholder flights page
- `/recommendation` shows the placeholder recommendation page

What it renders:
- The shared top navigation through `TabNav`
- The route content through `Routes` and `Route`

Why it matters:
- This is the file that binds the whole app together.
- If you add a new top-level section, it needs to be registered here.


File: `excel-haters/src/shared/TabNav.jsx`

Purpose:
- Shared navigation component for the top tab bar.

What it does:
- Receives a `tabs` array.
- Renders a `NavLink` for each tab.
- Applies active styling when the current route matches a tab route.

Why it matters:
- Keeps route navigation separate from the route declarations themselves.
- Makes the top bar reusable and easy to maintain.

5. CONFIGURATOR FEATURE
========================================================================

The configurator is the most complete part of the application.

It has two views:

1. Main dashboard view
- Shows Components and Terminals side by side

2. Terminal detail view
- Shows the selected terminal's components grouped into four type columns


-----------------------------------------------------------------------
5.1 Configurator pages
-----------------------------------------------------------------------

File: `excel-haters/src/configurator/ConfiguratorPage.jsx`

Purpose:
- Main dashboard page for the configurator tab.

What it imports:
- `ComponentTable`
- `TerminalTable`
- `configuratorContent`
- `useTerminalManager`

What it does:
- Calls `useTerminalManager()` to get:
  - local components data
  - fetched terminals data
  - loading state for terminals
  - error state for terminals
- Renders the configurator heading.
- Renders two side-by-side panels:
  - Component table on the left
  - Terminals table on the right

Why it matters:
- This is the landing page for the configurator feature.
- It coordinates data sources and passes them into the presentation components.


File: `excel-haters/src/configurator/TerminalComponentsPage.jsx`

Purpose:
- Detail page for one terminal.

Route:
- `/configurator/:terminalName`

What it imports:
- `useParams` and `Link` from React Router
- `ComponentTypeColumns`
- `useTerminalComponents`

What it does:
- Reads `terminalName` from the URL.
- Calls `useTerminalComponents(terminalName)`.
- Displays the current terminal title.
- Displays terminal metadata like type and active state.
- Displays the terminal's components grouped into four type columns.
- Provides a back link to the main configurator page.

Why it matters:
- It turns the terminal table into a drill-down workflow instead of a dead-end list.


-----------------------------------------------------------------------
5.2 Configurator components
-----------------------------------------------------------------------

File: `excel-haters/src/configurator/components/ComponentTable.jsx`

Purpose:
- Reusable tabular display for component records.

Expected data shape:
- `id`
- `name`
- `type`
- `isActive`

What it does:
- Validates component types against `componentTypeOptions`.
- Displays loading, error, empty, and normal states.
- Uses `useTablePagination` with a 20-row page size.
- Shows page number plus Previous and Next buttons.

Where it is used:
- Main configurator dashboard for the left-side component table.

Why it matters:
- It keeps the main component listing isolated from page-level logic.


File: `excel-haters/src/configurator/components/TerminalTable.jsx`

Purpose:
- Reusable tabular display for terminal records.

Expected data shape:
- `id`
- `name`
- `type`
- `isActive`

What it does:
- Validates terminal types against `terminalTypeOptions`.
- Displays loading, error, empty, and normal states.
- Uses `useTablePagination` with a 20-row page size.
- Adds a right-side Actions column.
- Renders a `View` link for each row.

The `View` link:
- Navigates to `/configurator/{terminalName}`
- Uses `encodeURIComponent` to safely encode spaces and special characters

Where it is used:
- Main configurator dashboard for the right-side terminals table.

Why it matters:
- It is the bridge from the main dashboard into the terminal detail page.


File: `excel-haters/src/configurator/components/ComponentTypeColumns.jsx`

Purpose:
- Specialized component used only on the terminal detail page.

What it does:
- Reads the allowed types from `componentTypeOptions`.
- Splits the components list into one column per type:
  - desk
  - security
  - gate
  - stand
- Shows the title and count for each column.
- Renders each component as a compact card-like row inside its type column.
- Shows active/inactive status for each component.
- Shows loading and error states if the data request has not completed.

Why it matters:
- This is the detail-page-specific presentation component.
- It replaces the single combined table with a grouped operational view.


-----------------------------------------------------------------------
5.3 Configurator hooks
-----------------------------------------------------------------------

File: `excel-haters/src/configurator/hooks/useTerminalManager.js`

Purpose:
- Main data-loading hook for the configurator dashboard.

What it imports:
- React state/effect hooks
- local `components.json`

What it fetches:
- `/api/terminals`

What it returns:
- `components`
- `terminals`
- `terminalsLoading`
- `terminalsError`

What normalization it does:
- Converts terminal `type` to lowercase.
- Forces `isActive` into a boolean.

Why it matters:
- It separates data-fetching logic from JSX rendering logic.
- It is the single source of truth for dashboard terminals state.


File: `excel-haters/src/configurator/hooks/useTerminalComponents.js`

Purpose:
- Data-loading hook for the terminal detail page.

What it does step by step:
1. Fetches `/api/terminals`
2. Normalizes the terminal list
3. Finds the terminal whose name matches the route parameter
4. Uses the matched terminal's ID to fetch:
   `/api/terminals/{id}/components`
5. Normalizes the component list

What it returns:
- `terminal`
- `components`
- `isLoading`
- `error`

Why it matters:
- The URL contains the terminal name, but the backend components endpoint needs
  the terminal ID.
- This hook resolves that mismatch.


File: `excel-haters/src/configurator/hooks/useTablePagination.js`

Purpose:
- Shared client-side pagination logic.

What it exports:
- `DEFAULT_PAGE_SIZE = 20`
- the `useTablePagination` hook itself

What the hook does:
- Tracks the current page number
- Calculates total pages from item count and page size
- Clamps the current page when the source data shrinks
- Returns the slice of items for the current page

Where it is used:
- `ComponentTable.jsx`
- `TerminalTable.jsx`

Why it matters:
- It keeps pagination logic out of the table JSX.
- Both tables behave consistently because they use the same hook.

7. HOW DATA CURRENTLY FLOWS THROUGH THE APP
========================================================================

Main configurator page flow:

1. Browser loads `index.html`
2. `main.jsx` mounts React
3. `App.jsx` chooses the `/configurator` route
4. `ConfiguratorPage.jsx` runs `useTerminalManager()`
5. `useTerminalManager()`:
   - loads components from local JSON
   - fetches terminals from `/api/terminals`
6. `ConfiguratorPage.jsx` renders:
   - `ComponentTable` on the left
   - `TerminalTable` on the right
7. `TerminalTable` paginates the terminals list and shows the `View` action


Terminal detail page flow:

1. User clicks `View` in `TerminalTable`
2. Router navigates to `/configurator/{terminalName}`
3. `App.jsx` matches `TerminalComponentsPage`
4. `TerminalComponentsPage.jsx` runs `useTerminalComponents(terminalName)`
5. `useTerminalComponents()`:
   - fetches `/api/terminals`
   - finds the terminal by name
   - fetches `/api/terminals/{id}/components`
6. `TerminalComponentsPage.jsx` renders `ComponentTypeColumns`
7. `ComponentTypeColumns.jsx` splits the fetched components into Desk, Security,
   Gate, and Stand columns

9. SHORT SUMMARY
========================================================================

If you want to understand this app quickly, these are the most important files:

1. `excel-haters/src/App.jsx`
   This tells you what pages exist.

2. `excel-haters/src/configurator/ConfiguratorPage.jsx`
   This is the main dashboard.

3. `excel-haters/src/configurator/hooks/useTerminalManager.js`
   This tells you where dashboard data comes from.

4. `excel-haters/src/configurator/TerminalComponentsPage.jsx`
   This is the terminal detail screen.

5. `excel-haters/src/configurator/hooks/useTerminalComponents.js`
   This tells you how the detail page resolves terminal name to terminal ID and
   then fetches components.

6. `excel-haters/src/index.css`
   This controls most of the visual system.

This means the current app is best understood as:

- a React Router shell
- one working feature area called Configurator
- API-backed terminals
- JSON-backed dashboard components
- route-based drill-down from terminals to terminal component columns
- placeholder pages for the other major tabs
